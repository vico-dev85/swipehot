## **Key Differences from Claude Code's Proposal**

### **1. Fastify instead of Express**
```typescript
// Express (what he suggested)
app.get('/api/next', (req, res) => {...})

// Fastify (what I recommend)
fastify.get('/api/next', async (request, reply) => {...})
```

**Why Fastify?**
- 2-3x faster than Express
- Built-in schema validation
- Better async/await support
- More modern architecture
- Still easy to learn

**Downside:** Slightly less popular than Express (but growing fast)

**Verdict:** ✅ Use Fastify if performance matters, Express if you want maximum community support

---

### **2. PostgreSQL instead of MySQL (Optional)**

**You already have MySQL on your server, so this is optional**

**Why PostgreSQL might be better:**
```sql
-- PostgreSQL handles JSON beautifully
SELECT * FROM performers 
WHERE tags @> '["latina", "bigboobs"]'::jsonb;

-- Better full-text search
SELECT * FROM blog_posts 
WHERE to_tsvector(content) @@ to_tsquery('cam & model');
```

**But if MySQL works for you:** Stick with it. Don't change unless you have a reason.

**Verdict:** ✅ MySQL is fine, PostgreSQL is slightly better for this use case

---

### **3. Next.js for Dashboard instead of pure React**

**Pure React SPA (what he suggested):**
```
dashboard/
├── index.html
├── bundle.js
└── Makes API calls to backend
```

**Next.js (what I recommend):**
```
dashboard/
├── pages/
│   ├── index.tsx (server-rendered)
│   ├── models.tsx
│   └── api/
│       └── config.ts (API routes in same app)
└── Built-in routing, SSR, optimizations
```

**Why Next.js?**
- Server-side rendering (faster initial load)
- Built-in routing
- API routes (can embed some API logic)
- Better developer experience
- Production-ready out of the box

**Downside:** Slightly more complex than pure React

**Verdict:** ✅ Next.js for dashboard, vanilla JS for frontend roulette

---

### **4. PM2 for Process Management**

**Must-have for production:**
```bash
# Start backend with PM2
pm2 start dist/server.js --name xcam-api

# Auto-restart on crash
# Cluster mode (use all CPU cores)
# Log management
# Zero-downtime deploys
```

**Verdict:** ✅ Essential for production

## **Monorepo Structure I Recommend**

```
xcam-platform/
├── packages/
│   ├── frontend/          # Roulette app (static HTML/CSS/JS)
│   ├── backend/           # API server (Fastify + TypeScript)
│   ├── dashboard/         # Admin CMS (Next.js)
│   ├── content-engine/    # SEO content generator (CLI)
│   └── shared/            # Shared types, utils, database models
├── database/
│   ├── migrations/        # SQL migrations
│   └── seeds/             # Test data
├── docker-compose.yml     # Optional: Redis, DB for local dev
├── package.json           # Root package.json (workspaces)
└── tsconfig.json          # Shared TypeScript config
```

**Why monorepo?**
- All TypeScript packages share types
- One `npm install` for everything
- Easy to refactor across packages
- Deploy independently but develop together
